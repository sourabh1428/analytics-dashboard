# Easibill — Honest Architecture, Product & UI Review
> Date: 2026-06-13 | Reviewer: Claude Code

---

## TL;DR

The core reminder pipeline (cron → BullMQ → Gupshup) is solid and correct. The backend
scaffolding is clean. But three things are genuinely broken today, the business model has a
quiet structural problem, and the UI has several rough edges that will confuse real pharmacists.
This doc covers all of it without sugarcoating.

---

## Part 1 — System Architecture

### What Is Working Well

- **BullMQ reminder pipeline** — `cronScheduler.ts` → `reminder-queue` → `reminderWorker.ts` is
  textbook-correct. 3 attempts, exponential backoff, `bull_job_id` stored for observability. Exactly
  as the PRD specifies.
- **Tenant scoping** — `tenantScope` middleware and `req.pharmacyId` used throughout. No
  cross-pharmacy data leaks found.
- **Migration discipline** — 10 sequential SQL migration files, run via `migrate.ts`. Solid
  foundation for a production DB.
- **Cloudflare Worker webhook forwarder** — a good pattern. Gupshup gets a stable public URL;
  the backend stays behind a VPC. Webhook delivery tracking (sent → delivered → read) is wired
  correctly.
- **Template variable mapper** — the `reminder_template_params` / `sale_template_params` system
  that maps `{{1}}`…`{{N}}` to named fields is genuinely well thought-out and matches how Gupshup
  template parameters actually work.

### Critical Bugs (Fix Before Any Customer Goes Live)

#### BUG-1: Broadcasts Bypass BullMQ — Rule #1 Violated

**File:** `backend/src/routes/broadcasts.ts:86`

```typescript
// Fire-and-forget: send to all recipients then update counts
(async () => {
  for (const patient of recipients) {
    await sendViaGupshup({ ... });
```

This calls Gupshup in a fire-and-forget loop directly from the route handler. There is no queue,
no retry, no backoff. If the server restarts mid-broadcast, half the patients get the message and
half don't, silently. The PRD's Rule #1 — "Every WhatsApp send goes through BullMQ" — is broken
here. Broadcasts are exactly the case that most needs a queue.

**Fix:** Add each broadcast recipient as a separate BullMQ job under a `broadcast-queue` (or
reuse `reminder-queue` with a `type` field). The route creates the broadcast row and enqueues
jobs; a worker processes them.

---

#### BUG-2: Sale Confirmation Also Bypasses BullMQ

**File:** `backend/src/routes/sales.ts:123`

```typescript
messageId = await sendTemplateViaGupshup({ to: waPhone, templateId: saleTemplateId, ... });
```

Same issue. The WhatsApp confirmation on sale completion is called inline from the POST handler.
If Gupshup has a momentary blip, the sale still completes but the confirmation is silently lost.
Even if it's "best-effort" (the comment says so), the failure is not recorded and there's no retry.

**Fix:** Enqueue a `sale-confirmation` job after committing the transaction. Worker handles the
send and logs the result.

---

#### BUG-3: Broadcast Tag Filter Queries the Wrong Table

**File:** `backend/src/routes/broadcasts.ts:53`

```sql
AND id IN (
  SELECT patient_id FROM patient_tags WHERE tag_id = $N
)
```

`patient_tags` is the **tag definition table** (pharmacy_id, name, color). It has no `patient_id`
column. The junction table is `patient_tag_map` (patient_id, tag_id). This query will always
return zero rows, so filtering by tag silently returns all patients — or throws a runtime error
depending on Postgres.

**Fix:** Change to `SELECT patient_id FROM patient_tag_map WHERE tag_id = $N`.

---

#### BUG-4: Timezone Bug — Cron Fires on UTC Date, Not IST Date

**File:** `backend/src/jobs/cronScheduler.ts:15`

```typescript
const today = new Date().toISOString().split('T')[0]; // UTC date string
```

`reminders.scheduled_for` is a `DATE` column representing a calendar date **in IST**. When the
cron fires at 03:20 UTC (= 08:50 IST), `new Date().toISOString()` returns the UTC date. From
18:30 IST (13:00 UTC) onward, the UTC date is already tomorrow. Any reminder created for
"tomorrow IST" could fire a day early, or not fire at all.

This is a latent bug — it will only manifest if `scheduled_for` dates are near a UTC midnight
crossing, but it's a known failure mode the PRD explicitly flagged.

**Fix:**
```typescript
const today = new Date().toLocaleString('en-CA', { timeZone: 'Asia/Kolkata' }).split(',')[0];
// or use date-fns-tz: formatInTimeZone(new Date(), 'Asia/Kolkata', 'yyyy-MM-dd')
```

### Architecture Drift from PRD

#### DRIFT-1: Baileys Starter Plan Never Built

The PRD's entire pricing model rests on:
- **Starter (₹299/mo)** — Baileys, pharmacy's own personal number, zero per-message cost
- **Pro (₹999/mo)** — Gupshup WABA, ₹1/message wallet credits

The code is **Gupshup-only for everyone**. There is no `baileysService.ts`, no QR code flow,
no Redis session. The `wa-connect` endpoint just flips a DB flag — it doesn't actually establish
any WhatsApp connection.

This is likely a conscious pivot (Baileys is notoriously unreliable), but it has consequences:
- Every pharmacy on the "Starter" plan is actually using Gupshup, which requires a 3–7 day BSP
  onboarding and has per-message costs
- The ₹299 plan is no longer viable at volume — a pharmacy sending 200 reminders/month would
  owe ₹200 in Gupshup fees on a ₹299 plan
- There's only one plan in practice, which means the pricing table in the PRD is fiction

**Decision needed:** Either rebuild Baileys (high risk), find a cheap alternative, or formally
merge the two plans and reprice.

#### DRIFT-2: `messageService.ts` Abstraction Missing

The PRD states: `messageService.ts` is the **"ONLY way to send WhatsApp messages."** It doesn't
exist. Instead, `sendViaGupshup` and `sendTemplateViaGupshup` are imported directly in:
- `routes/sales.ts`
- `routes/broadcasts.ts`
- `jobs/reminderWorker.ts`

When/if Baileys is ever added (or Gupshup is replaced), you'll need to update three files.

#### DRIFT-3: `broadcast_campaigns` Table is Dead

`001_initial.sql` creates `broadcast_campaigns`. Migration `007_broadcasts.sql` creates a
different, simpler `broadcasts` table. The code uses `broadcasts`. The `broadcast_campaigns`
table with its `filter_criteria JSONB`, `scheduled_at`, and `status` state machine is just sitting
in the schema doing nothing. Either remove it or replace the simple `broadcasts` table with it.

#### DRIFT-4: No Rate Limiting on Gupshup Sends

The PRD specifies 30 messages/hour per pharmacy number. No rate limiting is implemented anywhere.
For Gupshup WABA this matters less (the BSP handles throttling), but for broadcast sends that
loop through hundreds of patients, there's no delay or backoff between individual sends.

---

### Minor Architecture Notes

- `purchases` table in `001_initial.sql` doesn't have `unit_price`, `total_amount`, or
  `payment_method` — those are added in later migrations, which is fine, but the PRD schema
  doesn't mention them. Good additions — just undocumented.
- `message_logs.channel` check constraint in `001_initial.sql` is `CHECK (channel IN ('gupshup'))`
  — the `'baileys'` option was never added. Fix it or drop the check.
- The `is_test` column on `message_logs` doesn't exist in any migration file. It's referenced in
  `pharmacy.ts:115` and `sales.ts:143`. This will cause a runtime error until a migration adds it.
  Check if migration `005` or the unread migrations add it.

## Part 2 — Product (Principal PM Perspective)

### The One Metric That Matters Right Now

The PRD says: "Nothing else ships until this flow works end-to-end for 10 pharmacies."

**Status of the core flow:**
1. ✅ Pharmacist logs in
2. ✅ Pharmacist adds patient + records sale
3. ✅ System calculates refill date
4. ✅ BullMQ job scheduled
5. ⚠️ Cron fires at 8:50 AM IST (timezone bug, see BUG-4)
6. ✅ BullMQ worker sends via Gupshup template
7. ✅ Dashboard shows reminder status
8. ⚠️ WhatsApp "connected" check is cosmetic — no real connection test

The pipeline is functionally ~80% there. Fix BUG-4 and the `is_test` migration and the core flow
works. Don't ship more features until it's proven on real pharmacies.

---

### Missing Features That Block Revenue

#### PM-1: No Billing / Subscription Flow

`wallet_transactions` table exists. `wallet_credits` column exists on pharmacies. Nothing in the
frontend or backend touches them. There's no:
- Stripe/Razorpay integration for plan subscriptions
- Wallet top-up flow
- Low-balance alert (PRD specifies this at ₹50)
- Plan upgrade path from Settings

A paying customer can't pay. This blocks going from "free beta" to "paying subscription."

#### PM-2: Onboarding is Incomplete

`onboarding_step` column exists. An onboarding page exists at `/onboarding`. The dashboard shows
a 3-step checklist (Connect WhatsApp, Add Patient, Make Sale). But:
- Completing each step doesn't advance `onboarding_step`
- There's no redirect after onboarding completes
- The "Connect WhatsApp" step links to Settings but Settings → WhatsApp just shows a status
  badge — it doesn't guide the user through Gupshup BSP onboarding (which requires them to
  submit business documents externally)

A new pharmacist landing on day 1 has no guided path to first reminder. This kills activation rate.

#### PM-3: Patient Lapse Detection Exists in Backend but Not Surfaced

`reports/summary` returns `inactive_patients` (no purchase in 60 days). This is valuable.
But the Reports page isn't linked from the sidebar, and there's no proactive alert — no badge,
no dashboard card, no email. The PRD specifies lapse detection as a Pro feature. It's built
halfway and hidden.

#### PM-4: Broadcasts Have No Template — They'll Fail

The Broadcasts page lets a pharmacist type any free-text message. Gupshup WABA requires
pre-approved HSM templates for proactive messages (any message outside a 24h user-initiated
window). A free-text broadcast to 100 patients sent at any random time will be rejected by
Gupshup for all patients who haven't messaged the pharmacy number in the last 24 hours.

The UI gives no warning about this. The pharmacist thinks they sent a broadcast; Gupshup silently
fails most of them.

**Fix:** Either (a) restrict broadcasts to pre-approved templates like reminders do, or (b) add
a clear warning: "Broadcast messages only reach patients who have messaged you in the last 24
hours. To reach all patients, use a template."

#### PM-5: No Email Password Reset — It's Stubbed

`/auth/forgot-password` and `/auth/reset-password` routes exist. Frontend pages exist. But
`SMTP_HOST`/`SMTP_USER`/`SMTP_PASS` are likely not configured in production. If the email send
fails silently and the user can't reset their password, they're locked out permanently.
Verify this works before onboarding any customer.

#### PM-6: Data Export is "Coming Soon" — but PRD Doesn't Mention It

The Data tab in Settings has export buttons that toast "Coming soon." This is fine for now.
But if a pharmacist asks "can I get my patient list out," the answer is no. For compliance and
trust, this should be a Week 3 task, not a backlog item.

---

### Features to Deprioritize (Scope Creep Risk)

- **Households** — migration 006 added a `households` table. The PRD doesn't mention this at all.
  It's unused in any route. Remove or freeze it.
- **Multi-language templates beyond Hindi/English** — PRD says "Pro only." Don't build UI until
  the billing/plan split works.
- **Reports page** — the backend is mostly done but surfacing it with charts will take frontend
  time. The inactive patient list is the only thing that matters before 50 customers.

## Part 3 — UI/Designer Perspective

### What Is Working Well

- **Design system is consistent** — zinc neutral palette, `rounded-xl border border-zinc-200`
  cards, `text-sm font-semibold` section headers, Lucide icons. Clean and professional.
- **New Sale POS flow** is genuinely good — phone debounce → patient lookup → medicine
  autocomplete → single submit. Right interaction model for a counter.
- **Empty state + onboarding checklist** on dashboard is a nice touch for first-time users.
- **Inline editable fields** in Settings (pencil icon reveals input) is elegant.
- **Toast notifications** (Sonner) are well-placed and dismiss correctly.

---

### UI Bugs and Broken Patterns

#### UI-1: No Pagination — Will Break at Scale

Every list loads everything:
- `GET /patients` — no limit/offset in the route, no pagination UI
- `GET /reminders` — same
- `GET /broadcasts` — backend limits to 50, but UI shows all 50 with no paging

A pharmacy with 300 patients and 1,000 reminders will have a slow, janky page. Add
`page`/`limit` params to the routes and infinite scroll or "Load more" in the UI.

#### UI-2: Dashboard Data Goes Stale — No Polling or Refetch

```typescript
// page.tsx:31
useEffect(() => {
  Promise.all([...]).finally(() => setLoading(false));
}, []); // no deps, runs once
```

Stats, reminders, and activity are fetched once on mount and never updated. A pharmacist who
leaves the dashboard open all day will see yesterday's numbers. Add a 60-second polling interval
or a manual refresh button.

#### UI-3: Silent Error Swallowing on Dashboard

```typescript
.catch(() => {}) // page.tsx:42
```

If any of the four dashboard API calls fail (network blip, expired token, backend restart), the
page stays stuck in a spinner or shows zeros with no error message. The pharmacist doesn't know
if the data is real or broken.

**Fix:** Set an error state and show an inline banner: "Couldn't load dashboard data. Retry."

#### UI-4: Sidebar Doesn't Include Broadcasts or Reports

The sidebar has: Dashboard, New Sale, Patients, Reminders, Settings.

The dashboard quick actions link to `/broadcasts`. Settings links to `/reports`. But neither
`/broadcasts` nor `/reports` is in the sidebar nav. A pharmacist who navigates to Broadcasts
via the quick action button has no sidebar item highlighted, and no way to navigate back except
the browser back button.

**Fix:** Add Broadcasts and Reports to the sidebar. Reports can be hidden until 10 pharmacies
are live if needed.

#### UI-5: No Mobile Sidebar

```typescript
// layout.tsx — sidebar is fixed 220px
<aside className="fixed inset-y-0 left-0 w-[220px] ...">
```

No hamburger menu, no drawer, no responsive collapse. On a 375px mobile screen the main content
is under the sidebar. The PRD says "PWA via Next.js is sufficient for MVP" — but the current
layout isn't usable on mobile at all.

The primary user (pharmacist at counter) is on their phone 80% of the time.

**Minimum fix:** Add a hamburger button + sidebar drawer on screens below `md`.

#### UI-6: No Confirmation on Destructive Actions

- Deleting a patient: immediate — no confirm dialog
- Cancelling a reminder: immediate — no confirm dialog

A pharmacist who fat-fingers "Delete" on a patient with 3 years of purchase history loses that
history permanently (soft delete, but they don't know that). Standard UX requires a confirmation
dialog with the patient name displayed.

#### UI-7: Broadcasts Warning is Missing

As described in PM-4: the Broadcasts compose form has no warning that free-text messages only
work within a 24h session window. The UI should add a yellow callout explaining when template
vs. free-text messages are appropriate.

#### UI-8: "Connected" Badge is Misleading

In Settings → WhatsApp, the badge shows "Connected" if `wa_connected = true` in the DB. But
this flag is set by clicking a button — it doesn't verify that Gupshup can actually send messages.
A pharmacist who has `wa_connected = true` but an expired API key will see "Connected" and
then wonder why their reminders are all failing.

**Fix:** Use the `wa-test` endpoint result to determine badge state, not the DB flag. Or add a
"Verify" button that sends a test message and shows the real connection health.

#### UI-9: Settings "Team" and "Billing" Tabs Show Stub Cards

Both tabs render `<ComingSoon>` placeholder cards with no ETA or explanation. This creates the
impression the product is unfinished. Either:
- Remove these tabs from the nav until the features exist, or
- Show a "Join waitlist" / "Notify me" CTA with a brief description of what's coming

#### UI-10: Template Mapper Has No Validation Warning

If a pharmacist saves a template but doesn't map all the `{{N}}` variables, the reminder will
send with empty placeholder values (e.g., "Hello , your  is due."). The form shows the mapper but
there's no "you have unmapped variables" warning before saving. Add a yellow badge if any slot
has no selection.
